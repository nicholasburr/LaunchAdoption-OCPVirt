Infrastructure as Code:
Foundations

Mastering YAML and Git to treat your infrastructure like software.

Infrastructure as Code is a Mindset

  • Disaster Recovery: Rebuild lost environments in minutes, not days.
  • Auditability: Every infra change is tracked, approved, and reversible.
  • Scalability: Deploying 100 clusters takes the same effort as deploying one.

YAML: The Language of Configuration

What it IS

  • Data Serialization: Stores data in a human-readable format.
  • Declarative: Defines the desired state (the "what", not the "how").

What it ISN'T

  • No Logic: No variables, loops, or conditionals natively.
  • Not a Script: It doesn't "execute" actions on its own.

The Golden Rule: Indentation Matters

  • Consistency: Choose a standard indentation level across your team and stick to it.
  • Formatting: Maintain uniform spacing to ensure reliable parsing across platforms.
  • Alignment: Elements at the same hierarchical level must align vertically.

Git: The Core Workflow

1. Working Dir

Your local files. Where you actively create, edit, and test your code.

git add →
2. Staging Area

The prep zone. Intentionally select which specific changes go into the next save.

git commit →
3. Local Repo

Your local database. Saves a permanent historical "snapshot" of staged changes.

git push →
4. Remote Repo

The central server (e.g., GitHub). Acts as the team's shared source of truth.

No Auto-Saves: Git requires explicit actions. You must tell it what to stage, when to commit, and when to share.
